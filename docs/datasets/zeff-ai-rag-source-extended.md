# ZEFF AI RAG Source Extended

이 문서는 기존 RAG 원문에 handoff 문서 핵심, 대화 의사결정, Play Store 실무 메모를 더한 확장본입니다.
대화형 문맥과 운영 규칙을 함께 검색하도록 만들 때 이 파일을 우선 쓰면 됩니다.

## Chat Handoff 요약
- ZEFF AI는 https://zeffai.com 중심의 개인 AI 워크스페이스 웹앱입니다.
- Windows와 Android 앱은 독립 AI 프로그램이 아니라 웹앱을 여는 thin client 입니다.
- Windows 배포는 GitHub Releases의 unsigned installer 흐름을 유지합니다.
- Windows SmartScreen 경고는 정식 서명이나 Store 경로 없이 제거할 수 없습니다.
- 사용자는 비용 민감도를 우선했고, Windows 코드 서명 구매보다 Android + Play Store 경로를 선택했습니다.
- Android 패키지 ID는 com.zeffai.app 이고, production 진입 URL은 https://zeffai.com/login 입니다.
- Play Console 등록, 업로드 키스토어, workflow 복사, signed AAB, 실제 PLAY_STORE_URL 반영은 아직 남은 과업입니다.
- homepage는 큰 모니터에서 너무 작게 보이던 문제를 landing scale 설정으로 완화했습니다.
- mobile, tablet, desktop 스케일은 별도로 분리되어 있습니다.
- Prisma 7.8 환경에서는 prisma db push --skip-generate 를 쓰면 안 되고, build는 prisma generate && prisma db push && next build 형태를 유지해야 합니다.

## 이전 대화 핵심 의사결정
- 고비용 Windows OV/EV 인증서 구매는 현재 기본안이 아닙니다.
- 배포 신뢰를 가장 싸게 가져가려면 Android Play Store가 우선안입니다.
- Microsoft Store MSIX는 추후 Windows 신뢰 배포안으로만 보류합니다.
- 이미 결정된 방향은 다시 설득하지 않고 다음 실행 단계로 이어가는 것이 답변 원칙입니다.
- 기본 응답 언어는 한국어입니다.

## Play Store 제출 실무 메모
- Play 개발자 등록은 일반적으로 25달러 1회 비용입니다.
- 업로드 형식은 AAB 기준으로 준비합니다.
- 업로드 키스토어는 분실하지 않도록 안전하게 보관합니다.
- Play App Signing은 무료이며 최종 배포 키 관리를 Google에 맡길 수 있습니다.
- 공개 전에는 내부 테스트 트랙으로 먼저 설치와 로그인 흐름을 점검하는 편이 좋습니다.
- 개인정보처리방침 링크는 https://zeffai.com/support/legal#privacy 를 기준으로 씁니다.
- 공개 후에는 PLAY_STORE_URL 상수를 실제 스토어 링크로 바꾸고 랜딩 버튼을 연결합니다.
- assetlinks.json 의 SHA256 placeholder는 실제 서명 지문으로 교체해야 합니다.

## 답변 정책 메모
- ZEFF AI는 웹앱 중심 서비스로 설명합니다.
- 데스크톱과 모바일은 shell app 이라고 설명합니다.
- Groq는 추론 제공자로 설명하고, 학습 플랫폼처럼 단정하지 않습니다.
- SmartScreen 경고를 서명 없이 숨길 수 있다고 말하지 않습니다.
- 없는 기능을 구현 완료처럼 말하지 않습니다.
- 사용자가 비용을 중요하게 볼 때는 더 저렴하고 단순한 경로부터 제안합니다.

## 제품 개요
### ZEFF AI의 공식 도메인
- 핵심: 공식 도메인은 https://zeffai.com 입니다.
- 이유: 브랜드 링크와 배포 설명은 항상 이 주소를 기준으로 맞추는 편이 안전합니다.
- 실행: 문서와 버튼 링크도 이 도메인 기준으로 점검하면 됩니다.
- 주의: 테스트 도메인을 공식 주소처럼 안내하면 안 됩니다.
- 우선순위: 공식 링크 일관성 유지

### ZEFF AI의 핵심 제공 형태
- 핵심: 핵심은 웹앱이며, 데스크톱과 모바일은 그 웹앱을 여는 얇은 클라이언트입니다.
- 이유: 실제 기능과 데이터 처리는 클라우드 웹 서비스 중심으로 운영됩니다.
- 실행: 제품 소개 문구도 웹앱 중심 서비스로 맞추는 것이 좋습니다.
- 주의: 로컬 설치형 독립 AI처럼 설명하면 오해를 부를 수 있습니다.
- 우선순위: 웹앱 중심 설명 유지

### ZEFF AI의 서비스 구조
- 핵심: 현재 구조는 클라우드 기반 웹 서비스 중심입니다.
- 이유: 클라이언트 패키지 안에 AI 키를 넣지 않는 방향이기 때문입니다.
- 실행: 보안 설명에서는 서버 중심 처리 구조를 함께 안내하면 됩니다.
- 주의: 오프라인 완전 독립형으로 과장하면 안 됩니다.

### Windows 데스크톱 앱의 성격
- 핵심: Windows 앱은 Electron 기반의 얇은 셸 앱입니다.
- 이유: 실제 서비스는 https://zeffai.com 을 통해 제공됩니다.
- 실행: 기능 설명 시 웹앱과 동일한 계정 흐름을 안내하면 됩니다.
- 주의: 네이티브 로컬 AI 엔진처럼 설명하면 안 됩니다.

### Android 앱의 성격
- 핵심: Android 앱은 Capacitor 기반의 얇은 셸 앱입니다.
- 이유: production에서는 https://zeffai.com/login 경로를 로드합니다.
- 실행: 모바일 안내에서는 로그인 진입 경로를 함께 안내하면 좋습니다.
- 주의: 독립 실행형 오프라인 앱처럼 홍보하면 안 됩니다.

### 기본 응답 언어
- 핵심: 기본 응답 언어는 한국어로 유지하는 것이 맞습니다.
- 이유: 현재 사용자 선호와 기존 대화 흐름이 한국어 중심이기 때문입니다.
- 실행: 사용자가 언어를 바꾸면 그때 함께 전환하면 됩니다.

### 의사결정의 기본 기준
- 핵심: 의사결정에서는 비용 민감도를 우선 반영하는 편이 맞습니다.
- 이유: 배포 신뢰와 운영 효율 사이에서 저비용 경로를 선호하기 때문입니다.
- 실행: 대안을 제시할 때는 더 싸고 단순한 방법부터 보여주면 됩니다.
- 주의: 고비용 선택지를 기본안처럼 밀어붙이지 않는 것이 좋습니다.

### 이미 결정된 방향을 다루는 원칙
- 핵심: 이미 결정된 방향은 다시 설득하지 않고 그 안에서 다음 단계를 돕는 방식이 맞습니다.
- 이유: 사용자 선호를 존중하고 진행 속도를 유지할 수 있기 때문입니다.
- 실행: 질문이 오면 실행 항목과 체크리스트를 바로 제안하면 됩니다.

### 공식 로그인 진입점
- 핵심: 클라이언트 셸의 공식 로그인 진입점은 /login 경로입니다.
- 이유: 데스크톱과 모바일 모두 production origin 아래의 로그인 진입을 사용합니다.
- 실행: 셸 클라이언트 설명에는 /login 진입을 함께 표기하면 됩니다.

### 현재 운영 상태
- 핵심: ZEFF AI 웹 서비스는 현재 라이브 상태로 운영 중입니다.
- 이유: 공식 운영 주소는 https://zeffai.com 입니다.
- 실행: 배포 점검 시에는 실제 사이트 응답과 주요 로그인 흐름을 먼저 보면 됩니다.
- 상태: 웹은 라이브이고, Play Store 공개 링크는 아직 비어 있습니다.

## 배포 구조
### Windows 설치 파일 배포 방식
- 핵심: Windows 설치 파일은 GitHub Releases를 통해 배포하는 흐름입니다.
- 이유: 서명 인증서를 구매하지 않은 상태에서도 릴리스 파일을 배포할 수 있기 때문입니다.
- 실행: 릴리스 태그와 다운로드 링크를 함께 관리하면 됩니다.
- 주의: unsigned .exe에는 신뢰 경고가 따를 수 있습니다.

### Electron 앱의 실제 동작
- 핵심: Electron 앱은 내부에 AI 모델을 품는 대신 공식 웹앱을 로드합니다.
- 이유: 클라이언트에 AI 키를 넣지 않고 서버 중심으로 기능을 운영하려는 구조입니다.
- 실행: 설명 문구는 thin client 또는 shell app 기준으로 맞추면 됩니다.

### Android 패키지 ID
- 핵심: Android 패키지 ID는 com.zeffai.app 입니다.
- 실행: Play Console, asset links, 서명 설정에도 같은 ID를 일관되게 써야 합니다.
- 주의: 패키지 ID를 중간에 바꾸면 스토어 연속성이 깨질 수 있습니다.

### Android 앱의 로드 URL
- 핵심: Android 앱은 production에서 https://zeffai.com/login 을 로드합니다.
- 이유: 모바일 셸 앱도 웹앱 로그인 플로우를 그대로 사용하기 때문입니다.
- 실행: 테스트 시에는 로그인, 세션 유지, 외부 링크 이동을 같이 확인하면 됩니다.

### 현재 Play URL 상태
- 핵심: 현재 PLAY_STORE_URL 값은 비어 있어서 공개 스토어 링크가 아직 연결되지 않은 상태입니다.
- 실행: 공개 후에는 상수를 실제 스토어 URL로 교체하면 됩니다.
- 상태: Play Console 등록과 공개 링크 확정이 남아 있습니다.

### assetlinks.json 상태
- 핵심: assetlinks.json은 존재하지만 SHA256 지문은 아직 placeholder 상태입니다.
- 이유: 실제 앱 서명 지문이 준비된 뒤에 교체해야 하기 때문입니다.
- 실행: Play 업로드용 서명값이 확정되면 .well-known 파일을 업데이트하면 됩니다.
- 주의: placeholder 상태로 autoVerify를 기대하면 안 됩니다.

### Windows 빌드 워크플로 위치
- 핵심: Windows 설치 파일용 워크플로는 .github/workflows/build-windows-installer.yml 에 있습니다.
- 실행: 릴리스 자동화를 볼 때는 이 워크플로와 릴리스 태그 규칙을 같이 확인하면 됩니다.

### Android CI 템플릿 위치
- 핵심: Android CI 템플릿은 docs/ci/build-android.yml 에 두는 상태입니다.
- 이유: 이전 토큰에 workflow 스코프가 없어 .github/workflows 경로로 직접 푸시하지 못했기 때문입니다.
- 실행: 필요하면 GitHub 웹에서 내용을 복사해 실제 workflow 파일로 넣으면 됩니다.

### 클라이언트 패키지의 보안 원칙
- 핵심: 클라이언트 패키지에는 AI 키를 넣지 않는 방향이 맞습니다.
- 이유: 웹, 데스크톱, 모바일 모두 서버 중심으로 보안과 비용 제어를 하려는 구조이기 때문입니다.
- 주의: 클라이언트 번들 안에 비밀값을 넣으면 곧바로 노출 위험이 생깁니다.

### 배포 기준 브랜치
- 핵심: 배포 기준 브랜치는 main 입니다.
- 이유: Vercel이 GitHub의 main 브랜치를 기준으로 배포하는 흐름을 사용하기 때문입니다.
- 실행: 프로덕션 반영 전에는 main 기준 변경점과 빌드 상태를 확인하면 됩니다.

## Windows 서명과 신뢰
### Windows 코드 서명의 비용 구조
- 핵심: 공개 배포용 Windows 코드 서명 인증서는 일반적으로 비용 부담이 큰 편입니다.
- 이유: 신원 검증과 하드웨어 토큰 또는 HSM 요구가 붙기 때문입니다.
- 실행: 비용 민감한 경우에는 다른 배포 경로를 먼저 검토하는 편이 좋습니다.

### OV 인증서 비용대
- 핵심: OV 코드 서명 인증서는 대략 연 200달러에서 450달러 수준으로 안내되는 경우가 많습니다.
- 주의: 리셀러와 시점에 따라 실제 가격은 달라질 수 있습니다.

### EV 인증서 비용대
- 핵심: EV 코드 서명 인증서는 대략 연 250달러에서 700달러 이상으로 형성되는 경우가 많습니다.
- 이유: EV가 더 강한 신원 검증 절차를 요구하는 경우가 많기 때문입니다.
- 주의: 가격은 공급사와 정책 변화에 따라 달라질 수 있습니다.

### 하드웨어 토큰 요구
- 핵심: 최근에는 코드 서명에서 하드웨어 토큰이나 HSM 요구가 사실상 표준처럼 붙습니다.
- 이유: 키 유출 위험을 줄이려는 업계 정책 변화가 반영된 결과입니다.
- 주의: 단순 파일형 인증서만으로 끝난다고 기대하면 안 됩니다.

### 상대적으로 저렴한 대안
- 핵심: Azure Artifact Signing은 월 단위 과금으로 보일 수 있어 전통적 연간 인증서보다 저렴하게 느껴질 수 있습니다.
- 실행: 자격이 맞는지 먼저 확인한 뒤 비교하면 됩니다.
- 주의: 지역 제한이나 계정 조건을 반드시 확인해야 합니다.

### Microsoft Store 경로의 장점
- 핵심: Microsoft Store 경로는 별도 공인 코드 서명 인증서 구매 없이 신뢰 배포를 노릴 수 있는 대안입니다.
- 이유: 스토어 제출 시 Microsoft 쪽 재서명과 신뢰 경로를 활용할 수 있기 때문입니다.
- 실행: MSIX 패키징과 스토어 제출 절차를 함께 준비해야 합니다.

### unsigned .exe의 한계
- 핵심: 공개 배포용 unsigned .exe는 SmartScreen 경고를 피하기 어렵습니다.
- 실행: 이 제약을 전제로 다운로드 경로를 안내해야 합니다.
- 주의: 서명 없이 경고를 없앨 수 있다고 말하면 안 됩니다.

### EV 구매 판단 기준
- 핵심: SmartScreen만을 이유로 EV를 기본 추천하는 것은 현재 기준으로 효율이 낮을 수 있습니다.
- 이유: 예전처럼 즉시 우회 효과를 보장한다고 보기 어렵기 때문입니다.
- 우선순위: 비용 대비 효과 검토

### 전통적 인증서 결제 방식
- 핵심: 전통적 코드 서명 인증서는 보통 연 단위 선결제 구조입니다.
- 이유: 구독형이 아니라 연간 발급 및 검증 모델을 따르는 경우가 많기 때문입니다.

### Windows 대체 배포안
- 핵심: Windows에서는 MSIX와 Microsoft Store가 신뢰 배포 측면의 대체안이 될 수 있습니다.
- 실행: 필요하면 나중에 PC 경로는 MSIX, 모바일은 Play로 병행하는 전략을 검토하면 됩니다.
- 우선순위: 현재는 Android Play 우선

## Google Play
### Play 개발자 등록 비용
- 핵심: Google Play 개발자 등록은 일반적으로 25달러 1회 비용입니다.
- 이유: Windows 공인 코드 서명처럼 매년 인증서를 갱신하는 구조와는 다릅니다.

- 핵심: 사업자 등록이 확실하면 조직 계정이 좋고, 그렇지 않으면 개인 계정으로 시작하는 편이 현실적입니다.

- 핵심: D-U-N-S는 보통 조직 계정이나 기업 신원 확인 맥락에서 중요해질 수 있습니다.

## AI 제공자와 동작
### 지원하는 AI 제공자 수
- 핵심: 코드상 지원하는 AI 제공자 키 종류는 6개입니다.
- 이유: Gemini, Groq, Cerebras, Mistral, OpenRouter, DeepSeek를 기준으로 합니다.

### Gemini의 역할
- 핵심: Gemini는 텍스트뿐 아니라 이미지와 오디오 같은 멀티모달 처리에도 중요합니다.
- 이유: 현재 구현에서 파일 첨부 분석과 임베딩 폴백 전 상위 경로에 사용되기 때문입니다.

### Groq의 역할
- 핵심: Groq는 현재 이 프로젝트에서 빠른 추론 제공자 중 하나로 쓰입니다.
- 주의: 학습 플랫폼처럼 설명하지 않는 편이 맞습니다.

### OpenRouter의 역할
- 핵심: OpenRouter는 모델 그 자체라기보다 여러 모델을 연결하는 라우팅 계층에 가깝습니다.

### DeepSeek의 위치
- 핵심: DeepSeek는 초저가 추론 축으로 설명하는 편이 맞습니다.

### 다중 제공자 폴백
- 핵심: 현재 AI 호출은 여러 제공자를 후보로 두고 폴백하는 구조를 사용합니다.
- 이유: 한 제공자 실패 시 다른 후보로 넘어가 가용성을 높이기 위해서입니다.

### 키 기반 후보 필터링
- 핵심: 실제 호출 전에는 설정된 키가 있는 제공자만 후보로 남기는 필터링을 합니다.
- 이유: 없는 키를 가진 제공자를 먼저 때리지 않기 위한 구조입니다.

### 멀티모달 입력 조건
- 핵심: 현재 구현 기준으로 이미지나 오디오 분석에는 GEMINI_API_KEY가 특히 중요합니다.
- 주의: Gemini 키 없이 첨부 분석이 바로 된다고 말하면 안 됩니다.

### 임베딩 폴백
- 핵심: 임베딩은 GEMINI_API_KEY가 있으면 Gemini를 쓰고 없으면 로컬 임베딩으로 폴백합니다.
- 이유: 키가 없어도 기본 검색이 아예 막히지 않도록 하기 위한 설계입니다.

### AI 응답 스타일 원칙
- 핵심: 현재 방향에서는 한국어로 간결하고 실무적으로 답하는 스타일이 맞습니다.
- 실행: 결론 먼저, 이유는 짧은 목록으로 덧붙이는 형태가 적합합니다.

## RAG와 데이터 활용
### 문서 원문 저장 위치
- 핵심: 서재 문서의 분석 텍스트는 LibraryItem.extractedText 에 저장됩니다.
- 이유: 이 텍스트를 바탕으로 색인과 문서 기반 대화를 이어갈 수 있기 때문입니다.

### 색인 저장 구조
- 핵심: 색인된 조각은 DocumentChunk 테이블에 청크와 임베딩 형태로 저장됩니다.

### 청킹 방식
- 핵심: 청킹은 겹침이 있는 텍스트 분할 방식으로 문단과 문장 경계를 최대한 살리도록 되어 있습니다.

### 색인 API 입력
- 핵심: 색인 API는 libraryItemId 를 받아 해당 문서의 extractedText 를 청킹하고 임베딩합니다.

### 검색 상위 개수
- 핵심: 검색에서는 상위 관련 청크를 뽑아 컨텍스트로 묶어 답변에 사용합니다.
- 이유: 관련성이 높은 근거만 모델에 넣어 응답 정확도를 높이기 위한 구조입니다.

### 키 없는 검색 동작
- 핵심: GEMINI_API_KEY가 없어도 로컬 임베딩으로 기본 검색은 동작합니다.
- 주의: 품질은 외부 임베딩 대비 달라질 수 있습니다.

### 근거 표기 원칙
- 핵심: RAG 답변은 [1], [2] 같은 근거 번호를 붙여 설명하는 방식이 적합합니다.
- 이유: 문서 기반 응답의 신뢰성과 추적 가능성을 높일 수 있기 때문입니다.

### 보수적 답변 원칙
- 핵심: 문서에 없는 내용은 없다고 말하는 보수적 응답 원칙이 중요합니다.
- 주의: 컨텍스트 밖 내용을 지어내면 안 됩니다.

### grok 대화 활용법
- 핵심: grok 대화 데이터는 모델 재학습보다 문서로 정리해 RAG에 넣는 방식이 가장 현실적입니다.
- 이유: 현재 레포에 이미 색인과 검색 흐름이 있어서 빠르게 붙일 수 있기 때문입니다.

### 문서 분할 전략

### 과장 금지 정책
- 핵심: 없는 기능을 이미 구현된 것처럼 말하지 않는 것이 중요합니다.
- 주의: 배포 완료 전 기능을 공개 완료처럼 말하면 신뢰를 해칠 수 있습니다.

### SmartScreen 관련 금지 문구
- 핵심: unsigned .exe가 SmartScreen 경고 없이 설치된다고 말하면 안 됩니다.

### Prisma 금지 제안
- 핵심: 제거된 prisma db push --skip-generate 플래그를 다시 제안하면 안 됩니다.

### Play 출시 상태 설명 원칙
- 핵심: Play 등록이 끝나지 않았다면 공개 배포 완료처럼 설명하면 안 됩니다.

### 보안 답변 원칙
- 핵심: 민감한 키나 비밀번호를 평문으로 요구하거나 출력하지 않는 것이 중요합니다.

### 말투 원칙
- 핵심: 답변은 짧고 실무적이며 협업형 톤으로 유지하는 편이 좋습니다.
- 실행: 결론부터 말하고, 필요한 경우 이유를 2~4개 정도로 짧게 붙이면 됩니다.
